# Experimental FAF / BoundaryAttest handoff receipts

These are experimental FAF/BoundaryAttest mapping examples intended for external review by the FAF maintainer. They are not an official FAF integration, are not part of the stable BoundaryAttest interop profile, and are not stable verifier fixtures.

The proposed order is deliberately small:

1. add the opt-in MCP emitter first for write-, export-, and handoff-class boundary calls;
2. add the opt-in `faf-cli` emitter second for human or CI checkpoints such as export, sync, compile, or diff-log.

Both emitters use the same claim shape. `.faf`/`.fafm`, Git/file history, MCP traces, and `faf-cli` logs keep their existing responsibilities. BoundaryAttest signs only the selected handoff claim.

## Included receipts

- `mcp-write-receipt.json` represents an MCP `faf.write` result.
- `mcp-export-handoff-receipt.json` represents an MCP export/handoff and references the preceding receipt.
- `faf-cli-sync-checkpoint-receipt.json` represents the secondary CLI checkpoint and references the handoff receipt.

The JSON files are real Ed25519-signed receipts that verify with the existing BoundaryAttest example interop verifier and `faf-example-public-key.pem`. The key is generated by the repository's existing local example key setup. The public key included here identifies the signer used for the checked-in examples; distributing it with the examples does not make that signer trusted.

Running the generator re-signs and rewrites the example receipts with the machine's current local example key. It also rewrites the public-key file so the generated set remains self-consistent.

## Run it

From the repository root:

```sh
npm run build
npm run example:faf-mcp-handoff-v0.1
```

The command regenerates all three receipts, verifies each one, and exits unsuccessfully if any verification fails.

## Verification limits

Successful verification only proves that a specific signer signed a specific claim and that the signed claim was not altered. It does not prove the file/content is correct, the AI action was wise, authorization existed, or a business outcome occurred. It does not replace Git, FAF context, MCP traces, or `faf-cli` logs.

The claim values and referenced artifacts are synthetic. These examples demonstrate a possible mapping and signature binding; they are not evidence that the represented FAF or MCP operations occurred.

See the [mapping note](../../docs/faf-boundaryattest-mapping-v0.1-draft.md) for field meanings and relying-party limits.
